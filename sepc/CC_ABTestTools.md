# 设计目标
* 针对Dify平台，基于dify版本 设计一套可以AB Test不同版本效果，并给到人评估的工具；用于单一workflow变更后的效果评测。

# 适用范围与场景：
* 基于dify workflow的版本以及历史数据 进行回测，评估模型效果，包括：
	- Auto Eval（主要涵盖输出的稳定性，结构化结果的合法性，workflow的响应时长 做整体评估）
	- Preference Labeling（基于输出内容 让业务人员标注偏好版本）

# 结果可视化：
* 基于测试任务对不同版本 workflow 的Auto Eval 和 Preference Labeling 做可视化展示

# 整体工具的工作流程
1. 设置实验版本信息
- 需要设置包括：
	a. 实验workflow版本信息：
		- 支持设置一个对照组（一般为现有方案）和多个实验组（参与实验的workflow，最多2个实验组)的信息包括：
			i. workflow 的 api Key （一次实验限制配置一个）
			ii. workflow 的 workflow_id
			iii. 输入参数的json结构
			iv. 输出参数的json schema
			v. response_mode（默认为blocking，支持选择streaming)，一次测试仅支持设置同样的请求方式。
			vi. 版本名称与信息，支持flag版本 以及 版本差异化内容
			Vii. 输出结果合并规则，参考python的f string规则，支持将输出参数的json schema 中的部分参数 设置合并规则（可选，若开启Preference Labeling，必填）
	b. 实验的评估目标：
		- Auto Eval（默认）：
			+ 成功率：包括成功响应的 、 输出报文中有报错的、超时响应 任务占总体任务占比
			+ 稳定性：输出符合json schema的占所有成功的任务的占比
			+ 响应时长：统计所有成功响应任务的响应时长的平均值，中位数、上下四分位值，单位为秒
				* response_mode使用blocking，则需要统计从请求发起 到收到响应结果后的时长
				* response_mode使用streaming，则需要统计从请求发起，到收到workflow_finished 事件为止
				* 不统计 返回失败，超时的 和 报文中有错误的任务
		- Preference Labeling（可选，默认不开启）
			+ 开启后，支持将测试的结果数据 根据“输出结果合并规则” 合并输出内容后，将结果输出到标注平台，支持标注人员选择版本

	c. 实验数据：
		+ 支持通过json List导入测试数据
			- 导入时，先根据输入参数的json schema 校验数据是否合法；存在不合法数据 需要报错提醒
			- 校验通过后，将导入的数据 分别根据设置的对照组和实验组 进行遍历执行，并获取和存储结果
2. 实验结果观测：
* 图表规则
	- 成功率和输出结构稳定性：采用柱状图展示对照/实验组的率和占比
	- 响应时长：采用箱型图展示对照/实验组的响应时长平均、中文和上下四分位值
	- Preference Labeling 展示2组数据
		+ 已经标注的数据占比所有进入标注的数据占比，采用环形图展示
		+ 用柱状图表示对照/实验组的采纳数量
	- 统一数据标注颜色：所有图表颜色规范如下
		+ 对照组：#0A84FF
		+ 实验组A：#FF375F
		+ 实验组B：#6750A4
		+ 容器内字体标注均采用 #FFFFFF
	- 统一精度：
		+ 成功率、输出结构稳定性 精度为 0.01%
	- 图表规范：
		+ 需要展示图例（颜色和实验组名）
		+ 柱状图容器内展示数据
		+ 环形图中间展示数据
		+ 箱型图右侧标注数据
* 实验状态机
	- 创建实验后，若为导入数据，则实验为「未开始」状态
	- 导入数据并校验通过后，实验为「进行中」状态
		+ 若为开启“Preference Labeling”，则当数据完整遍历所有对照和实验组，则任务为「完成」状态，此时可以查看实验结果数据
		+ 若开启“Preference Labeling”，则当数据完整遍历所有对照和实验组，任务为「进行中」状态，此时可以查看Auto Eval的数据，“Preference Labeling”支持实时查看数据；任务状态直到所有数据标注完成后手动关闭实验后，任务状态为「完成」

3. 账号管理
* 账号采用RBAC管理，角色包括2种
	- 标注人员，限制访问标注任务列表，以及详情，可以进行标注；无法能看到其他页面
	- 管理人员，有系统所有页面的访问以及操作权限
* 账号元素：
	- 账号和密码（2个字符串组，均支持标点符号和英文，不支持中文）组合用于登录
	- 账号数据采用json 进行简单维护（人员较少）
	- 密码需要md5加密存储

4. Preference Labeling标注数据的进入机制：
	- 所有的对照组/实验组中，需要均成功返回结果，才进入标注数据，有任意缺失均不进入标注数据


# 交互设计
## 页面
1. 登录页面，支持账号密码登录，并对账号密码进行校验；
 - 若登录账号为管理员角色，则进入实验列表页面
 - 若账号为管理员角色，则进入标注列表页面

2. 实验列表页面
- 页面本身展示所有的实验
	+ 列表展示信息包括：
		* 实验名称
		* 实验创建时间
		* 实验状态（见状态机）
	+ 列表支持操作：
		- 详情，点击后，当前页面跳转到查看实验详情
		- 查看实验报告，点击后进入 实验报告页面
		- 导入数据，点击后弹出Modal，支持导入csv文件
	+ 列表排序规则：按实验创建时间 逆序排列
	+ 全局操作：
		- 创建实验，点击后进入实验详情页面
	+ 操作与状态机关系：
		- 「未开始」 支持 详情，导入数据 操作
		- 「进行中」支持 查看实验报告 操作
		- 「完成」支持 查看实验报告 操作

2.1. 实验详情页：
- 支持编辑和创建实验信息，包括：
	* 实验基础信息
		- 实验名称，样式：input，最长50个汉字
		- workflow.API_Key，样式input，最长150个字母符号
		- response_mode，默认为blocking，支持选择“streaming”，样式为radio
		- Preference Labeling，样式switch ，默认关闭
		- 输入参数的 json-schema
	* 对照组信息
		- 版本名称，样式：input，最长50个汉字；用于图表中的图例名称
		- 版本描述，样式：input.textField，最长300个汉字
		- workflow_id，样式input，最长150个字母符号
		- 输出参数的 json-schema
		- 开启「Preference Labeling」后，支持设置「输出结果合并规则」，样式：input.textField，最长1000个字母符号，做基本的规则校验
	* 实验组信息，默认设置1个，最多支持设置2个实验组，包括：
		- 版本名称，样式：input，最长50个汉字；用于图表中的图例名称
		- 版本描述，样式：input.textField，最长300个汉字
		- workflow_id，样式input，最长150个字母符号
		- 输出参数的 json-schema
		- 开启「Preference Labeling」后，支持设置「输出结果合并规则」，样式：input.textField，最长1000个字母符号，做基本的规则校验
	* 页面级校验规则：
		- 实验基础信息中，实验名称，workflow.API_Key 页面非空校验
		- 对照组信息中，版本名称、workflow_id、输入参数的 json-schema、输出参数的 json-schema 均需要做非空校验；
		- 实验组信息中，版本名称、workflow_id、输入参数的 json-schema、输出参数的 json-schema 均需要做非空校验；
	* 错误反馈：
		- 提供非空校验反馈，采用Disabled Submit Pattern
		- 对于输入参数的 json-schema，输出参数的 json-schema 做json-schema的合法校验，在焦点离开输入框是进行，若不符合则采用Inline Validation + Progressive Enablement 提示非法输入
		- 对于「输出结果合并规则」 同样进行DSL的合法性校验，若不符合则采用Inline Validation + Progressive Enablement 提示非法输入

- 页面布局
	* 采用整页卡片式布局，顶部固定提交和取消按钮
	* 卡片分为基础信息 和实验组信息卡片
		- 实验组信息卡片下 再分为 对照组卡片、实验组A卡片和实验组B卡片

2.2. 实验详情
- 页面布局：采用整页卡片式布局
	+ Auto Eval 卡片，展示固定的评估信息
	+ 若开启「Preference Labeling」，则展示Preference Labeling数据
	+ 空值处理：若无数据 则需要提示“暂无数据”提示

3. 标注列表页面
- 页面展示所有开启「Preference Labeling」的实验信息，包括：
	* 实验名称
	* 实验创建时间
	* 标注进度
		+ 展示已标注数据条数
		+ 总计的数据条数
- 列表支持操作：
	* 查看标注数据，点击后进入标注数据列表

3.1. 标注数据列表
- 页面展示对应实验的所有标注数据信息，包括：
	* 标注任务的UUID
	* 标注状态
		- 已标注
		- 未标注
	* 对应标注胜出版本信息（取标注的版本内容）
- 列表支持操作：
	* 标注，点击后，通过右侧抽屉展示标注详情

3.1.1. 抽屉-标注详情页
- 页面布局采用卡片纵向布置，并按固定顺序（从左到右）展示对照组、实验组A和实验组B的结果信息，每个卡片需要展示信息包括：
	* 版本名称
	* 版本描述（若无则隐藏）
	* 对应的workflow_id
	* 输出结果，采用input.textField,展示内容为根据输出结果以及输出结果合并规则后的效果，需要支持对Markdown进行渲染（主要为最多3级title，换行，加粗、斜体样式）
- 卡片整体为radio，支持点击选择；默认不选择任何卡片
- 底部固定为
	* 提交按钮，对是否选择卡片做校验，采用Disabled Submit Pattern；点击后提交标注数据
	* 取消按钮，关闭抽屉

# 技术要求
## 开发语言
- 后端采用python，前端服务需要通过fastAPI拉起
- 前端采用TypeScript，并使用ant design的框架
## 账号管理
- 需要支持cli创建账号
- 账号密码需要MD5加密后存储到json文件
## 数据存储
- 实验数据需要存储到数据库
## 调用与并发
- 对dify不做并发调用，需要逐个调用。避免超过dify worker上限导致请求超时而引发Auto Eval的失真
## 日志管理
- dify调用请求、标注数据、以及实验的创建更新都需要存储到日志
- 日志采用json格式，并用kv存储，记录信息需要包括Who，What，When,where,Why, trace级别，参考：
"""
{
  "timestamp":"2026-06-23T14:21:01Z",
  "level":"INFO",
  "service":"user-service",
  "user_id":"12345",
  "action":"login",
  "status":"success"
}
"""
